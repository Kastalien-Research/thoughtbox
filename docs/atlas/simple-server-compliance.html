Executive Summary

Simple Server is a production Ruby on Rails application serving as the backend for Simple.org, a hypertension and diabetes management platform deployed across India, Bangladesh, Ethiopia, and Sri Lanka. It stores highly sensitive protected health information (PHI) including patient names, dates of birth, phone numbers, addresses, blood pressure and blood sugar measurements, diagnoses, prescription drugs, and treatment history for hundreds of thousands of patients.

Overall HIPAA Risk Rating
HIGH-MEDIUM

The application demonstrates a mature engineering culture with strong fundamentals: comprehensive RBAC, SSL enforcement, parameter filtering, audit logging infrastructure, and a well-structured test suite. However, several gaps represent genuine HIPAA compliance risks in a formal HHS audit context, particularly around encryption at rest for individual PHI fields, session timeout enforcement, account lockout configuration for admin users, content security policy deployment mode, and cross-country data segregation controls. No PHI is exposed in plaintext logs due to explicit filtering, which is a significant positive. The DORA metrics profile shows a High performance team with deployment frequency approaching Elite territory.

  • Confirmation gate before production and staging deploys (prevents accidental pushes)
  • Sentry deployment notifications (sentry:notice_deployment after publish)
  • Database migrations run via db:migrate:with_data (data migrations included)
  • Passenger web server restart with sudo
  • Sidekiq worker management (4 processes)
  • Whenever/cron schedule updated on deploy

Feature Flag Usage (Flipper)

The codebase uses Flipper for feature flags stored in ActiveRecord, enabling trunk-based development where incomplete features can be shipped to production gated behind flags. This is a DORA best practice:

actor.respond_to?(:power_user?) && actor.power_user?

Feature flags used across codebase: :notifications, :experiment, :fixed_otp, :auto_approve_users, :drug_stocks, and many more. This enables gradual rollout and instant kill-switches.

Migration Safety

Database migrations use the data_migrate gem which enables data migrations alongside schema migrations. A review of recent migrations shows:

  • Primarily use def change (reversible) — good practice
  • View updates (Scenic gem) create database views as migrations
  • Partitioned table migrations for reporting data
  • One known long-running migration caused a failing deployment: 834ed84 fixing failing deployment due to long running migration
  • Foreign key removal (20260212195326_unlink_patients_from_cvd_risks.rb) is irreversible without a compensating migration
db/migrate/20260212195326_unlink_patients_from_cvd_risks.rb
class UnlinkPatientsFromCvdRisks < ActiveRecord::Migration[6.1]
def change
remove_foreign_key :cvd_risks, :patients # irreversible
end
end
Lead time is estimated at 1–5 days based on PR merge to deployment. The manual confirmation gate is a deliberate safety choice for a medical system but does reduce deployment frequency. Evidence of same-day hotfixes shows the team can move fast when needed.

High
No Log Retention Policy or Integrity Controls
Audit logs are flat files in log/audit.log and log/patient_lookup_audit.log on the application server. No WORM storage, cryptographic signing, or immutable log shipping is configured. HIPAA Technical Safeguards — §164.312(c) Integrity

Control Status Evidence / Notes
Data validation on PHI fields PASS Patient model validates: status presence, date_of_birth (no future dates), associated address and phone numbers. PhoneNumberAuthentication validates phone uniqueness. MedicalHistory enforces one-way enum rules (preventing downgrade from yes/no to suspected). Extensive validation across all models.
Sync conflict resolution PASS Device-timestamped conflict resolution via Mergeable concern: compares device_updated_at, keeps newer record (last-write-wins). Handles discarded/invalid/identical states. Idempotent by UUID — same record merged multiple times produces same result. app/models/concerns/mergeable.rb
Medical history integrity rules PASS One-way enum enforcement silently prevents degrading a confirmed diagnosis (yes/no) back to suspected. Date rules prevent htn_diagnosed_at from being changed once set. app/models/medical_history.rb:69-113. This is a meaningful clinical data integrity control.
Strong parameters / mass assignment PASS All API endpoints use explicit params.require().permit() with allowlists. Patient sync enumerates every permitted field explicitly. app/controllers/api/v3/patients_controller.rb:67-124
Database constraints on PHI tables PASS PostgreSQL foreign keys, NOT NULL constraints (enforced by structure.sql), and unique constraints on phone numbers and UUIDs ensure referential integrity at the DB level.
Soft delete integrity PARTIAL The discard gem provides soft-delete. Patient#discard_data soft-deletes patient and all associated records in a transaction. However, soft-deleted records remain in the database and are syncable (scope :for_sync, -> { with_discarded }). Soft-deleted patient data is still accessible via raw database query.
Idempotency / duplicate prevention PASS UUID-based idempotency for all sync operations. Sidekiq unique jobs (sidekiq-unique-jobs 7.1) prevent duplicate background job execution. Patient deduplication service handles duplicate passport identifiers.
Medium
SQL Interpolation Risk in Patient Age Scope
The where_current_age scope in Patient model directly interpolates a comparison_operator parameter into a raw SQL string. While the age parameter is passed as a separate integer, the comparison operator is not sanitized. If this scope is ever called with user-provided input for the comparison operator, it would create a SQL injection vulnerability.
app/models/patient.rb:112-118

where("EXTRACT(YEAR FROM ...) #{comparison_operator} #{age}")
# comparison_operator is interpolated — not parameterized
end

HIPAA Technical Safeguards — §164.312(e) Transmission Security

Control Status Evidence / Notes
HTTPS enforcement PASS config.force_ssl = true in production.rb (line 51). This enforces SSL for all connections, adds HSTS headers, and redirects HTTP to HTTPS. Strong control.
SMTP transport security PASS SendGrid SMTP configured with enable_starttls_auto: true and authentication: :plain over port 587. config/environments/production.rb:79-87

XSS attacks would succeed despite CSP configuration. This is a significant weakness. config/initializers/content_security_policy.rb:26

Patient name in SMS is indirect PHI disclosure to the mobile carrier. This is inherent to appointment reminder use cases but should be documented as a risk acceptance.

High
Content Security Policy in Report-Only Mode
The Content Security Policy is configured as report_only = true, which means it is purely advisory. XSS attacks against the admin dashboard will succeed because the browser does not block any script execution. Given that the admin dashboard renders patient PHI, this is a meaningful risk. The CSP also allows unsafe_inline for both scripts and styles, which would need to be addressed before enforcement is enabled.

Medium
Patient Name in SMS Appointment Reminders
SMS notifications include patient_name in the message body (app/models/notification.rb:47-53). This transmits patient PHI (name + appointment date + facility name) via SMS, which is transmitted through carrier networks without encryption. This is a known and accepted risk for appointment reminder systems, but should be formally documented as a risk acceptance with patient consent (reminder_consent field — good, this is checked). Encryption at Rest

Critical
No Application-Level Encryption of PHI Fields
There is no encrypted column gem (attr_encrypted, lockbox, blind_index) in the Gemfile. Patient PHI including names, dates of birth, phone numbers, addresses, and medical history is stored in plaintext PostgreSQL columns. Database-level encryption (Transparent Data Encryption) cannot be verified from the codebase alone, but is not configured at the application level. For a HIPAA-regulated system, encryption at rest for PHI is a Required Implementation Specification under §164.312(a)(2)(iv) when the relevant risk analysis determines it is a reasonable safeguard.

This means: if the PostgreSQL database is compromised (backup theft, misconfigured backups, insider threat), ALL patient data is immediately readable without any additional decryption step.
Gemfile — no encryption gem found
# No attr_encrypted, lockbox, blind_index, or similar
# PHI columns stored in plaintext PostgreSQL
# Example PHI fields (patient.rb associations):
full_name, date_of_birth, age, gender # Patient table
street_address, village_or_colony, district, state # Address table
number (phone_number) # PatientPhoneNumber table

  • Mobile passwords: bcrypt via has_secure_password — PASS
  • Access tokens: random 64-character hex, stored hashed/plain — PARTIAL
  • Rails secrets: encrypted via read_encrypted_secrets = true — PASS
  • PHI fields: plaintext in PostgreSQL — FAIL
  • Backup encryption: not verifiable from codebase — UNKNOWN
  • Doorkeeper OAuth tokens: stored unhashed (hash_token_secrets not enabled) — FAIL

Doorkeeper OAuth access tokens are stored in plaintext in the database (hash_token_secrets not enabled). If the oauth_access_tokens table is compromised, all valid access tokens are exposed and can be used to make authenticated API requests. The Doorkeeper configuration explicitly comments out hash_token_secrets as a possibility but does not enable it. Used by machine users for the DHIS2 integration and other server-to-server authentication.

Data Deletion and Right to Erasure

The Patient#discard_data method provides soft-deletion of all patient-associated records in a single transaction. This covers: address, appointments, blood pressures, blood sugars, business identifiers, observations, encounters, medical history, phone numbers, prescription drugs, teleconsultations. However:

  • Soft-delete (discard) leaves data in the database — data is not actually deleted
  • Audit logs referencing patient IDs remain
  • Materialized views may cache patient data after soft-delete
  • No documented data retention or hard-delete capability for regulatory "right to erasure" requests

fixed_otp Feature Flag

Critical
fixed_otp Feature Flag Creates Authentication Vulnerability
The :fixed_otp Flipper flag causes OTP generation to always produce "000000" (sampling only [0]). This is clearly a development convenience feature but if accidentally enabled in production it would allow any attacker to authenticate as any mobile user by knowing their phone number and using OTP "000000". The flag must be verified to be disabled in all production environments.

# If :fixed_otp enabled, OTP is always "000000"
otp = ""
6.times { otp += digits.sample.to_s }
end
