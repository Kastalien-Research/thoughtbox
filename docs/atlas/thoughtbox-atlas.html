Codebase Overview

Key Insights
Central Assembly Point
GOLD
src/server-factory.ts (2,067 lines) is the single assembly file that imports all 18+ modules and wires them into the MCP server. Every handler, tool, resource, and capability is registered here.
Progressive Disclosure State Machine
GOLD
ToolRegistry manages 4 disclosure stages (S0 Entry → S1 Init → S2 Cipher → S3 Domain). Tools are enabled/disabled dynamically as the workflow advances. Domain filtering at S3 enables context-specific tool sets.
Gateway as Single Entry Point
GOLD
The always-enabled thoughtbox_gateway tool routes all 30+ operations internally. Solves the MCP client tool-list refresh problem by never changing the visible tool count — all routing is internal.
Known P1 Bug: ESM/CJS Export
P1 BUG
src/persistence/export.ts has a broken export path due to ESM/CJS module mismatch. Session export functionality (exportSession) fails at runtime. All other persistence operations unaffected.
Hub: Full Multi-Agent Protocol
GOLD
Hub implements a complete agent coordination protocol: identity/profiles, workspaces, problems (with sub-problems + dependencies), proposals (with reviews), consensus markers, and real-time channels. 36 test files, 117+ test cases.
Observatory: Real-Time WebSocket
GOLD
Observatory provides a fire-and-forget thoughtEmitter that publishes thought events over WebSocket channels. Three channel types: reasoning, observatory (meta), and workspace. Includes bundled HTML UI at src/observatory/ui/observatory.html.

Architectural Patterns
Key recurring design patterns found across the codebase. These patterns are consistently applied and define the system's architectural DNA.
Pattern 1: Gateway Routing (Single Tool, Internal Dispatch)
Instead of exposing many MCP tools (which causes client tool-list refresh churn), the server exposes exactly ONE always-enabled tool (thoughtbox_gateway) and routes all operations internally. New operations are added to the schema enum — no tool count changes.
// gateway-handler.ts — the core routing pattern export const gatewayToolInputSchema = z.object({ operation: z.enum([ 'get_state', 'navigate', 'start_new', 'cipher', 'thought', 'notebook', 'session', 'mental_models', 'knowledge', 'hub', 'read_thoughts', 'get_structure', 'deep_analysis', // ... 20+ more operations, all routed through ONE tool ]), args: z.record(z.any()).optional(), }); // server-factory.ts — single tool registration server.tool('thoughtbox_gateway', GATEWAY_DESCRIPTION, gatewayToolInputSchema, async (input) => gatewayHandler.handle(input) );
Pattern 2: Progressive Disclosure State Machine
The ToolRegistry implements a one-way state machine with 4 stages. Tools register with an enabledAtStage declaration and are automatically enabled/disabled as the session advances. Stage 3 adds domain filtering — only tools matching the activeDomain are enabled.
export enum DisclosureStage { STAGE_0_ENTRY = "entry", // only: init operations STAGE_1_INIT_COMPLETE = "init_complete", // adds: cipher, session STAGE_2_CIPHER_LOADED = "cipher_loaded", // adds: thought, notebook, mm STAGE_3_DOMAIN_ACTIVE = "domain_active", // adds: domain-filtered tools } // Tools self-declare their activation stage: registry.register('thoughtbox', tool, DisclosureStage.STAGE_2_CIPHER_LOADED, { [DisclosureStage.STAGE_2_CIPHER_LOADED]: "Full reasoning with branching..." }); // Stage advancement is one-way (no regressions): if (targetIdx < currentIdx) { console.warn("Ignoring backward stage transition"); return; }
Pattern 3: Storage Abstraction + Doubly-Linked Thought Graph
The ThoughtboxStorage interface decouples all persistence from implementation. Thoughts are stored as a doubly-linked tree (not just an array) to support branching, revision chains, and sub-problem hierarchies. Each node has prev, next[], revisesNode, and branchOrigin pointers. Content hashing uses a Merkle chain for integrity.
interface ThoughtNode { id: ThoughtNodeId; // "{sessionId}:{thoughtNumber}" data: ThoughtData; prev: ThoughtNodeId | null; next: ThoughtNodeId[]; // array enables branching (tree not list) revisesNode: ThoughtNodeId | null; branchOrigin: ThoughtNodeId | null; branchId: string | null; // Merkle chain: contentHash?: string; // SHA-256(content+thoughtNumber+parentHash+agentId+ts) parentHash?: string; }
Pattern 4: Fire-and-Forget Event Emission (Observatory)
The thoughtEmitter singleton uses a fire-and-forget pattern — events are emitted synchronously but listener failures are completely isolated. Reasoning performance is never degraded by monitoring. The emitter checks hasListeners() before constructing payloads.
// In ThoughtHandler.processThought(): if (thoughtEmitter.hasListeners()) { thoughtEmitter.emitThoughtAdded({ sessionId: this.currentSessionId, thought: observatoryThought, parentId: previousThought?.id ?? null, }); // No await — never blocks the reasoning path } // Observatory WebSocket server routes by channel: // ws://localhost:1731/reasoning → thought stream // ws://localhost:1731/observatory → meta/improvement events // ws://localhost:1731/workspace → hub collaboration events
Pattern 5: Operations Catalog Pattern
Each domain module (gateway, init, sessions, knowledge, hub) exports a getOperationsCatalog() and getOperation(name) pair. This allows the gateway handler to dynamically look up operation handlers without compile-time coupling to every operation implementation.
// server-factory.ts imports all catalogs: import { getOperationsCatalog as getGatewayOps, getOperation as getGwOp } from "./gateway/operations.js"; import { getOperationsCatalog as getInitOps, getOperation as getInitOp } from "./init/operations.js"; import { getOperation as getHubOp } from "./hub/operations.js"; // ... 6 domain catalogs total // GatewayHandler routes at runtime: const op = getGwOp(input.operation) ?? getInitOp(input.operation) ?? ...; return op.handle(input.args, context);
Pattern 6: Hub Progressive Disclosure (3 stages within Hub)
Hub has its own 3-stage disclosure separate from the global ToolRegistry stages. Stage 0 allows only identity and workspace discovery. Stage 1 adds workspace creation and profile access. Stage 2 unlocks full problem/proposal/consensus/channel operations. This layering prevents information overload for new agents.
// hub-types.ts — Hub's internal 3-stage system: export const STAGE_OPERATIONS: Record<DisclosureStage, HubOperation[]> = { 0: ['register', 'list_workspaces', 'quick_join'], 1: ['whoami', 'create_workspace', 'join_workspace', 'get_profile_prompt'], 2: [ 'create_problem', 'claim_problem', 'list_problems', 'create_proposal', 'review_proposal', 'merge_proposal', 'mark_consensus', 'post_message', 'workspace_digest', // ... 15+ more operations ], };

Evolution Timeline
Development milestones inferred from commit history, CHANGELOG.md, proof-run findings, and implementation summaries. The project shows a clear arc from single-tool reasoning server to full multi-agent coordination platform.
Early 2025
v1 — Sequential Thinking MCP
Original single-tool ThoughtHandler with basic sequential thought chains. InMemoryStorage only. No branching or revision support. Spawned from "sequential-thinking" MCP pattern.
Jan 2025
Persistence Layer + FileSystem Storage
Added SQLite metadata + JSON file storage. Monthly time-partitioned session directories. Doubly-linked ThoughtNode graph for branching support. Merkle chain content hashing for integrity.
Jan 2025
Progressive Disclosure (ToolRegistry + 4 Stages)
Introduced ToolRegistry with STAGE_0 through STAGE_3. Solved MCP client tool-list refresh problem. Gateway tool pattern established as primary API surface. This is the architectural inflection point.
Jan 2025
Observatory + WebSocket Monitoring
Added real-time WebSocket server with fire-and-forget ThoughtEmitter. Three channel types (reasoning/observatory/workspace). Bundled HTML UI. Prometheus metrics endpoint for Grafana integration.
Jan 2025
Hub: Multi-Agent Coordination Protocol
Major feature: full multi-agent protocol with AgentIdentity, Workspaces, Problems (DAG dependencies), Proposals (review lifecycle), Consensus markers, and real-time Channels. 36 test files. Defined in ADR-002.
Jan 2025
Phase 3: Autonomous Critique (Sampling Loops)
SamplingHandler added for autonomous self-critique. When critique:true is set in a thought, the server calls MCP createMessage API to get model feedback and stores it back on the thought.
Jan 2025
Evaluation Pipeline (LangSmith, Phases 1-3)
Added LangSmith integration with dataset management, experiment runner, online monitor, and 4 evaluator types (reasoning coherence, memory quality, session quality, DGM fitness). ScorecardAggregator for trend tracking.
Jan-Feb 2025
Mental Models (15 frameworks) + Knowledge Graph
Added 15 reasoning framework resources. Added Entity/Relation/Observation knowledge graph for cross-session memory. Distinct from ephemeral thought sessions.
Feb 2025
Docker + OTel + Prometheus stack
Dockerfile with native better-sqlite3 compilation. Docker Compose with Prometheus + Grafana. OTel sidecar. Fixed /tmp permissions for OTel. Observatory sidecar for external consumers.
Feb 2025 (current)
Active: Gitignore cleanup + loops-content restoration
Stopped blanket-ignoring .claude/ directory. Restored loops-content.ts from HEAD after accidental deletion. Known open issues: P1 ESM/CJS export bug in persistence, multi-agent cipher limited integration, demo port mismatch.
Planned
DGM Self-Improvement Loop
Darwin Growth Machine: autonomous improvement cycles where the server modifies itself based on evaluation scores. dgm-specs/ harness is functional; hypothesis and target directories are placeholder stubs. self-improvement/ contains planning docs.
