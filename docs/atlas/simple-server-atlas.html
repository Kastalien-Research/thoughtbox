Medical Backend at Scale

Simple Server is a production Rails 7 API powering hypertension and diabetes care across India, Bangladesh, Sri Lanka, Ethiopia, and beyond. It serves mobile health workers who record patient vitals offline and sync to a central server — making data integrity, conflict resolution, and offline-first sync the architectural core of everything.

Key Architectural Challenges
Offline-first mobile sync. Health workers record data offline on Android devices. The bidirectional sync protocol (GET/POST /sync) must handle conflict resolution, out-of-order delivery, and partial syncs across unstable connections.
Multi-country deployment. Country configs (YAML) drive locale, supported genders, SMS providers, calendar systems (Ethiopia), and feature flags per region. Same codebase, dozens of deployment contexts.
Materialized reporting at scale. Patient population analytics use PostgreSQL ltree, partitioned tables, and materialized views refreshed on schedule — not real-time queries — to serve dashboards to program managers.
Patient deduplication. Patients registered at multiple facilities need merging without losing clinical history. The deduplication pipeline is a recurring hotspot.

Model Dependency Graph

Patient is the gravity well of this system. Every clinical record — blood pressures, blood sugars, prescriptions, appointments, encounters — belongs to a Patient. The Facility and Region models form the structural backbone through which access control, reporting scope, and mobile sync are all scoped.

class Patient < ApplicationRecord
  include Mergeable        # conflict resolution
  include Hashable         # anonymized exports
  include PatientReportable

  # Clinical records
  has_many :blood_pressures
  has_many :blood_sugars
  has_many :prescription_drugs
  has_one  :medical_history
  has_many :encounters
  has_many :observations, through: :encounters

  # Scheduling
  has_many :appointments
  has_many :notifications
  has_many :call_results

  # Teleconsult
  has_many :teleconsultations

  # Identity
  has_many :phone_numbers
  has_many :business_identifiers
  belongs_to :address, optional: true

  # Facilities (dual: registration + assigned)
  belongs_to :registration_facility
  belongs_to :assigned_facility

  # Experimentation
  has_many :treatment_group_memberships
  has_many :treatment_groups
  has_many :experiments
end
Sync Concern — Mergeable
# The key concern for offline-first sync.
# Every syncable model includes this.
# Handles: conflict resolution, soft deletes,
# device timestamps vs server timestamps.

module Mergeable
  # device_created_at: when recorded on device
  # device_updated_at: last update on device
  # recorded_at:       clinical timestamp
  # deleted_at:        soft delete (Discard)
Sync protocol design: Clients POST local records with device timestamps. Server resolves conflicts using device_updated_at — last writer wins per record. GET /sync returns records changed since a process_token (cursor-based pagination).

Sync Protocol — How It Works
Upload Flow (Device → Server)
1
Device collects records offline
Blood pressures, appointments, patients written to local SQLite while offline
2
POST /patients/sync
Batch of records sent with device_created_at, device_updated_at, UUID identifiers
3
MergePatientService resolves conflicts
Last device_updated_at wins. Soft-deleted records preserved with deleted_at.
4
Audit log written (async)
CreateAuditLogsWorker queued in Sidekiq for compliance tracking
Download Flow (Server → Device)
1
GET /patients/sync?limit=100&process_token=...
Cursor-based pagination using opaque process_token (encoded timestamp)
2
Scope by facility region
Returns patients registered or assigned within the health worker's block — not all patients
3
Transforms + serializes
app/transformers/api/ converts Rails model to versioned API payload format
4
Returns next process_token
Client continues paging until empty result set, then considers sync complete
Historical sync tolerance (added 2026-01): sc-17557 added the ability to accept invalid app data in historical sync APIs — an acknowledgment that real-world devices send malformed records and the server must be resilient rather than rejecting data from patients' clinical histories.
API v1 + v2 are gone. Both namespaces return HTTP 410 Gone for all routes. The codebase is explicit: match "*all", via: [:get, :post, :put, :delete, :patch], to: proc { [410, {}, [""]] }. Maintains clean version history without dead code accumulation.

The dominant architectural shift in this window: PostgreSQL materialized views are being replaced by partitioned tables across the reporting layer. Materialized views require a full refresh and are heavyweight. Partitioned tables allow incremental inserts and faster range queries. This refactor touches structure.sql, reporting.rake, schedule.rb, and multiple report model files simultaneously — explaining why those files cluster at the top of the hotspot list.
L4 · Patterns

Key Architectural Patterns

Simple Server uses a consistent set of architectural patterns that reflect the constraints of deploying healthcare software in low-resource environments: unreliable connectivity, multi-country contexts, data quality challenges, and the need for program managers to see meaningful health outcome metrics without SQL access.

🔄
Bidirectional Cursor-Based Sync
Every syncable resource implements GET + POST at /sync. Upload uses device UUIDs and timestamps to identify records. Download uses an opaque process_token cursor for pagination. Conflict resolution is last-write-wins on device_updated_at. This design handles offline-first mobile clients that may sync days after recording. The Mergeable concern provides the conflict resolution logic. The Transformers::Api and Validators::Api layers handle versioned serialization and input validation cleanly separated from the domain model.
app/controllers/api/v3/sync_controller.rb app/models/concerns/mergeable.rb app/transformers/api/ app/validators/api/
🌳
PostgreSQL ltree for Organizational Hierarchy
The Region model uses PostgreSQL's ltree extension to store organizational paths as strings (e.g., india.punjab.amritsar). This enables efficient ancestor/descendant queries using standard ltree operators, drives all access control decisions (admins see their subtree), and scopes mobile sync (health workers download their block's patients). Every report is parameterized on a Region path prefix. FriendlyId provides human-readable slugs for URLs.
app/models/region.rb app/models/facility_group.rb app/services/region_tree_service.rb db/structure.sql (ltree extension)
📈
Query Object Pattern
Complex analytical SQL is extracted into dedicated Query Objects in app/queries/. Each encapsulates a specific question: Which patients are overdue? (OverduePatientsQuery), What are the cohort follow-up rates? (CohortAnalyticsQuery), How many follow-ups at each facility? (FacilityAnalyticsQuery). This keeps models clean and queries testable in isolation.
app/queries/overdue_patients_query.rb app/queries/cohort_analytics_query.rb app/queries/follow_ups_query.rb app/queries/district_analytics_query.rb
🗄
Materialized Views + Partitioned Tables for Analytics
Program manager dashboards cannot run analytical queries in real-time across millions of patient records. The solution: pre-compute everything. The Reports::Refreshable concern marks models backed by materialized views or partitioned tables. A Sidekiq job (RefreshReportingPartitionedTableJob) and Rake task (reporting.rake) refresh them on schedule. Currently mid-migration from materialized views (heavyweight, full refresh) to partitioned tables (incremental, query-efficient).
app/models/reports/refreshable.rb app/services/refresh_reporting_views.rb app/jobs/refresh_reporting_partitioned_table_job.rb lib/tasks/reporting.rake
🧪
Feature Flags via Flipper (Per-Region)
The Flipperable concern exposes Flipper feature flags to the Region model. Features can be enabled for specific organizations, states, or facilities. This is how multi-country deployment is managed: diabetes management, experimental SMS providers, Dr. Rai reports, and teleconsultation are all flag-gated. The admin can enable features for Bangladesh without affecting India.
app/models/concerns/flipperable.rb app/helpers/flipper_helper.rb config/initializers/flipper.rb /flipper (Flipper::UI mounted)
🔀
Multi-Provider SMS Channel Abstraction
SMS delivery is abstracted behind a Messaging::Channel interface with four concrete implementations: Twilio (global), Alpha SMS (Bangladesh), BSNL (India), Mobitel (Sri Lanka). The NotificationDispatchService selects the right provider based on the patient's assigned facility region. Each provider has its own delivery detail model for tracking. Experimenting on which SMS content improves appointment adherence is handled by the Experimentation module.
app/services/messaging/channel.rb app/services/messaging/twilio/ app/services/messaging/alpha_sms/ app/services/notification_dispatch_service.rb
🔍
Patient Deduplication Pipeline
Patients registered at multiple facilities with slightly different names or phone numbers create duplicate records. The deduplication pipeline in app/services/patient_deduplication/ identifies likely duplicates using passport IDs (BP passport QR codes) and phone number matching. MergePatientService merges clinical histories without data loss. A DeduplicationLog tracks merged pairs. The admin panel exposes a manual deduplication review UI. This is a recurring source of hotspot activity because real-world data is messy.
app/services/patient_deduplication/ app/services/merge_patient_service.rb app/models/deduplication_log.rb app/controllers/admin/deduplicate_patients_controller.rb
📊
ViewComponent for Dashboard Widgets
The program manager dashboard uses GitHub's ViewComponent library for encapsulated UI components in app/components/. The Dr. Rai report widget, diabetes registrations chart, and progress tab are each independent ViewComponents with their own Ruby class and ERB template. This replaces Rails partials with testable, encapsulated view logic.
app/components/application_component.rb app/components/dashboard/dr_rai_report.html.erb app/components/reports/ app/components/progress_tab/
Notable Technical Debt Signals
structure.sql is the bottleneck. Every schema change regenerates the entire file (11 commits in 50). All developers see merge conflicts in this file constantly. The PostgreSQL-specific features make schema.rb impossible, but the file is still a coordination overhead.
Parallel materialized view + partitioned table world. Both architectures coexist during the migration. Some report models still use mat views, others use partitioned tables. The Refreshable concern must handle both, creating dual-path complexity until the migration completes.
Multi-version API maintenance cost. v3 and v4 overlap significantly. Both have their own controllers, transformers, and validators for many of the same resources (patients, appointments, sync). This duplication means changes to the data model must often be reflected in two controllers.
Data migration files accumulate. db/data/ contains 27 data migration files, many for specific country SMS experiment setups. These are idempotent but accumulate over time, representing historical operational state rather than schema evolution.
