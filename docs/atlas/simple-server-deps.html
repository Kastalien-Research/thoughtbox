Concerns, Mixins & Cross-Model Behaviors
Mergeable
Patient, Facility, BloodPressure, BloodSugar, Appointment, Address, PatientPhoneNumber, PatientBusinessIdentifier, PrescriptionDrug, MedicalHistory, Teleconsultation, Communication, CallResult, CvdRisk, PatientAttribute, DrugStock — implements CRDT-style merge protocol: new/updated/identical/old/discarded/invalid states. Enables conflict-free mobile sync.
Hashable
Patient, BloodPressure, Appointment, Communication — provides hash_uuid() for anonymized data exports. Ensures PHI-stripped exports via deterministic UUID hashing.
Flipperable
User, Region — adds feature_enabled?(:flag_name) instance method delegating to Flipper. Enables per-user and per-region feature gating (e.g., drug_stocks, auto_approve_users, exotel_whitelist_api).
PatientReportable
Patient — adds with_hypertension, with_diabetes scopes. Base for all cohort/follow-up analytics queries. Critical for reporting pipeline.
PatientReportable (Matview)
MaterializedPatientSummary, BloodPressuresPerFacilityPerDay, PatientRegistrationsPerDayPerFacility — extends PatientReportable for materialized view models. Adds Refreshable for CONCURRENT REFRESH support.
Mergeable
The Mergeable protocol implements device_updated_at comparison: if incoming timestamp > existing → update; equal → identical (no-op); less → old (keep existing). Critical for multi-device offline sync.
PhoneNumberLocalizable
PatientPhoneNumber, PhoneNumberAuthentication — provides localized_phone_number using Phonelib gem + country ISD codes. Renders phone numbers in country-appropriate format for SMS delivery.
Observeable
BloodPressure, BloodSugar — manages has_one :observation polymorphic relationship. Ensures clinical readings are always tied to an Encounter via Observation join table.
BloodPressureable
Provides BP-related scopes (critical?, hypertensive?) — used by Patient#risk_priority calculation and Appointment eligibility checks.
LiberalEnum
Facility — allows enum values outside the defined set without raising errors. Handles legacy data and API schema evolution gracefully.
DiagnosedConfirmedAtSync
Patient — syncs htn_diagnosed_at and dm_diagnosed_at from MedicalHistory during device sync. Handles confirmed diagnosis timestamps for HTN/DM.
SQLHelpers
BloodPressure (extended) — provides class-level SQL helper methods for complex raw queries. Used in analytics and reporting.
RegionSource
Facility (extended), FacilityGroup (extended) — adds has_one :region, inverse_of: :source, foreign_key: "source_id". Wires model instances to their Region node in the ltree hierarchy.
Reports::Refreshable
MaterializedPatientSummary, BloodPressuresPerFacilityPerDay, PatientRegistrationsPerDayPerFacility, Reports::View — provides .refresh! class method for REFRESH MATERIALIZED VIEW CONCURRENTLY.
Reports::RegionCaching
Reports::Repository — cache key generation and cache busting for region-based report caching. Works with BustCache concern.
DrRai::Calculatable
DrRai data models — provides calculation methods for DR-RAI (AI-assisted district performance indicators).
DrRai::Chartable
DrRai::Data::BpFudging, DrRai::Data::Statin, DrRai::Data::Titration — chart data extraction for DR-RAI dashboards.

Materialized Views — Dependency Order
The following views must be refreshed in dependency order (RefreshReportingViews enforces this):

Tier 1 (raw data, no view deps): reporting_patient_blood_pressures reporting_patient_blood_sugars reporting_prescriptions
Tier 2 (depends on Tier 1): reporting_patient_visits reporting_patient_follow_ups
Tier 3 (depends on Tier 2): reporting_facility_states reporting_overdue_patients reporting_overdue_calls
Tier 4 (depends on Tier 3): reporting_quarterly_facility_states reporting_facility_appointment_scheduled_days
Regular Views (no data, computed on query): reporting_facilities patient_summaries reporting_months reporting_facility_monthly_follow_ups_and_registrations
Partitioned Tables (simple_reporting schema): reporting_patient_states reporting_facility_monthly_follow_ups_and_registrations reporting_patient_prescriptions — Monthly partitions via PostgreSQL RANGE partitioning. add_shard_to_table procedure creates new partitions.

Cross-Cutting Concerns
Sync Protocol (Offline-First Mobile)
Models implementing Mergeable (sync-capable):
Patient · Address · PatientPhoneNumber · PatientBusinessIdentifier · BloodPressure · BloodSugar · Appointment · MedicalHistory · PrescriptionDrug · Encounter · Teleconsultation · Communication · CallResult · CvdRisk · PatientAttribute · DrugStock · Facility

Merge decision matrix:
• device_updated_at > existing → UPDATE
• device_updated_at == existing → IDENTICAL (no-op)
• device_updated_at < existing → OLD (keep server)
• existing.discarded? → DISCARDED
• new_record.invalid? → INVALID

Sync scope: Block-level sync (district_level_sync? for teleconsult users), facility-level for others
Authorization (UserAccess + RegionAccess)
Access levels (UserAccess::LEVELS):
power_user → full access (cross-org)
manager → facility_group level
viewer → read-only reports
call_center → overdue patients only

Access resource types (polymorphic):
Organization · FacilityGroup · Facility

RegionAccess: Memoized (Memery) per-request region tree access check. Builds accessible_region_ids per region_type. Used in: accessible_facilities, accessible_district_regions, etc.

Delegated from User: accessible_organizations, accessible_facilities, accessible_facility_groups, accessible_district_regions, accessible_block_regions, accessible_facility_regions, accessible_users, accessible_admins, can_access?, grant_access
Feature Flags (Flipper)
Known feature flags:
auto_approve_users — skips manual approval for new users
drug_stocks — enables drug stock tracking per district
exotel_whitelist_api — enables Exotel phone whitelisting
dr_rai — AI-assisted performance indicator dashboards
:power_users — group flag for power user actors

Stored in: flipper_features + flipper_gates tables (ActiveRecord adapter)

Usage pattern: Region/User include Flipperable → feature_enabled?(:flag) → Flipper[:flag].enabled?(self)
Soft Delete (Discard gem)
Pattern: ApplicationRecord includes Discard::Model
All models have deleted_at column. Default scope excludes discarded.

with_discarded: Used in sync scopes — mobile app needs deleted records to know what to remove locally

Cascade discard: Patient#discard_data cascades to all associated records in a transaction (HIPAA-style deletion):
address, appointments, blood_pressures, blood_sugars, business_identifiers, observations, encounters, medical_history, phone_numbers, prescription_drugs, teleconsultations

Discard callbacks: Region#after_discard cascades to estimated_population; User#after_discard destroys email_authentications
Caching Architecture
Layers:
1. PostgreSQL materialized views — pre-computed aggregates, refreshed nightly
2. Rails.cache (Redis) — region report data, ancestor lookups
3. Memery — per-object instance memoization (request-scoped)
4. Cache-warmer job — RegionCacheWarmerJob pre-populates region report cache

Cache keys: Region#cache_key = "regions/region_type/id", Region#cache_version = updated_at.utc. Cached ancestors use path + version.

BustCache concern: Reports::Repository uses BustCache to invalidate on region updates.
Audit Logging
AuditLog model: Records all sync operations — creates, updates, deletes per model.

CreateAuditLogsWorker: Async bulk insert using activerecord-import for performance.

PatientLookupAuditLogJob: Compliance log for patient data access events.

Request context: request_store + request_store-sidekiq propagates current_user to job context for audit attribution.

CPHC migration audit: CphcMigrationAuditLog tracks per-record migration status (polymorphic) for NHM compliance.
A/B Experimentation Framework
Models: Experimentation::Experiment → TreatmentGroup → TreatmentGroupMembership ← Patient

Types:
• CurrentPatientExperiment — patients with recent visits
• StalePatientExperiment — patients not seen recently
• NotificationsExperiment — full notification content/timing experiments

Runner: Experimentation::Runner enrols patients into treatment groups, creates Notifications per ReminderTemplate. Filter by states/blocks/facility_slugs.

Production shadow: Reports::Repository uses Scientist gem to run new/old code paths side-by-side in production (scientific A/B for backend query changes)
