Context: CyDoc is a 2009-era Swiss doctor's office EMR application for patient records, TARMED billing, and insurance integration. Swiss law applies DSG/nDSG (Federal Data Protection Act), KVG (health insurance), and UVG (accident insurance) rather than HIPAA. This report applies HIPAA as a reference standard for technical best-practice evaluation of medical data handling. All findings should be mapped to applicable Swiss law before acting. The codebase was last actively maintained in 2014–2016 and targets Rails 3.2 — a version with no security support since 2013.
⚠
Would Not Pass a Medical Compliance Audit
The application stores substantial Protected Health Information with no encryption at rest, a hardcoded secret token committed to git, no dedicated audit logging of PHI access, session timeout disabled by default, account lockout unconfigured, and a Rails version with zero security support since June 2013. DORA performance is "Low" across all four metrics, reflecting a project in maintenance-only mode with no CI pipeline and manual deployment via Capistrano.

Critical Findings

Would Fail a Real Audit
Critical Secret Token Committed to Git
The Rails secret_token (used to sign all session cookies) is hardcoded in config/initializers/secret_token.rb and present in git history. Any party who has ever cloned this repository can forge valid session cookies and impersonate any user — including administrators — with full access to all patient data.
# config/initializers/secret_token.rb CyDoc::Application.config.secret_token = '2b3ab0cce593906df7b9913f235be9aa9142cd12...'
The token cannot be considered secret. It must be rotated immediately and moved to an environment variable, invalidating all current sessions.
Critical Rails 3.2 — Zero Security Support Since 2013
The application runs Rails 3.2.19 (locked in Gemfile.lock). Rails 3.x reached end-of-life in June 2013. Over a decade of known CVEs — including SQL injection, remote code execution, and XSS vulnerabilities — are unpatched. This is a critical compliance failure for any application storing medical records.
actionpack (3.2.19) # EOL 2013-06 activerecord (3.2.19) # 10+ years of unpatched CVEs cancan (1.6.8) # Succeeded by cancancan bcrypt-ruby (3.0.1) # Old version
Critical No Encryption at Rest for Any PHI
Every sensitive field — patient names, birth dates, diagnoses, insurance numbers, treatment records, clinical notes — is stored in plaintext in the MySQL database. Uploaded attachments (lab results, medical documents) are stored in plaintext on the local filesystem. A database backup file or disk image provides immediate access to all patient health information with no cryptographic barrier.
Critical Session Timeout Disabled
Despite declaring :timeoutable in the Devise strategy list, the timeout duration is commented out in config/initializers/devise.rb. This means authenticated sessions never expire from inactivity — a workstation left logged in is permanently accessible. HIPAA § 164.312(a)(2)(iii) explicitly requires automatic logoff.
# config/initializers/devise.rb # config.timeout_in = 30.minutes <-- NEVER SET
Critical No Audit Log of PHI Access
HIPAA § 164.312(b) requires audit controls that record and examine activity in information systems containing PHI. No audit logging gem (PaperTrail, Audited) is present. No custom logging of record access is implemented in any controller or model. PHI reads are completely untracked — there is no way to determine who accessed which patient records.
Critical No Account Lockout — Brute Force Possible
Devise :lockable is listed in the User model strategies but the lockout policy is entirely commented out in the configuration file. There is no limit on failed login attempts. An attacker can attempt unlimited password guesses against any known username without triggering any account protection mechanism.
# devise.rb — all lockable config commented out: # config.lock_strategy = :failed_attempts # config.maximum_attempts = 20 # config.unlock_in = 1.hour
High Cross-Tenant Patient Data Exposure in Search
app/controllers/search_controller.rb inherits from ApplicationController, not AuthorizedController. It calls Patient.by_text(query) and Case.where(:praxistar_eingangsnr => code) without any tenant scope. In a multi-tenant deployment, a user from one practice could search and view patients belonging to another practice.
class SearchController < ApplicationController def index Patient.by_text(query, ...) # No .where(tenant: current_tenant) Case.where(:praxistar_eingangsnr => code) # No tenant filter end end
High PHI Leakage into Application Logs
Rails default parameter logging will record patient names, birth dates, diagnosis codes, and insurance numbers passed in URL parameters or POST bodies into log/production.log. These logs are stored in plaintext with no access controls beyond filesystem permissions. The TARMED billing search (app/controllers/search_controller.rb) and patient search pass PHI through params[:query] which is logged.
High No Role Differentiation — All Admins See Everything
The Ability class grants can :manage, :all to both admin and sysadmin roles, with no other roles defined. A receptionist booking appointments has identical data access to the treating physician. The minimum necessary access standard (HIPAA § 164.514(d)) is not implemented at any level.

Improvement Recommendations

Ordered by Risk Priority
1
Rotate the Secret Token and Remove from Version Control
Immediately replace the hardcoded secret_token in config/initializers/secret_token.rb with an environment variable (ENV['SECRET_TOKEN']). Rewrite git history or treat the old token as permanently compromised. This invalidates all existing user sessions — acceptable as it closes the session-forgery attack vector. Do not store secrets in any tracked file.
1
Enable Session Timeout
Uncomment config.timeout_in = 30.minutes in config/initializers/devise.rb. This is a one-line change that immediately satisfies HIPAA § 164.312(a)(2)(iii). Consider 15 minutes for a clinical environment where workstations may be shared. Also enable cookie :secure flag.
1
Enable Account Lockout
Uncomment and configure the Devise lockable strategy in config/initializers/devise.rb: config.lock_strategy = :failed_attempts, config.maximum_attempts = 10, config.unlock_strategy = :email. This prevents brute-force attacks against provider credentials.
1
Add Audit Logging for PHI Access
Add PaperTrail or the Audited gem and enable it on Patient, Treatment, MedicalCase, Diagnosis, and ServiceRecord models. Store whodunnit (current user ID) for every create/update/destroy event. Additionally, log read access to patient records in a separate audit table (PaperTrail does not log reads — implement a before_action callback for this).
2
Fix Cross-Tenant Exposure in SearchController
Either move SearchController to inherit from AuthorizedController and add tenant scoping to all queries, or add explicit where(doctor_id: current_tenant.id) (or equivalent) scopes to Patient.by_text and the Case lookup. Also add an authorization check to verify the found Case belongs to the current tenant.
2
Add Role-Based Access Control with Minimum Necessary Privilege
Expand the Ability class in app/models/ability.rb to define at least three roles: admin (full access), physician (access to own patients only), and receptionist (appointments, recalls, billing only — no clinical data). This is the most architecturally significant change and should be planned carefully, but it is required for any reasonable compliance posture.
2
Filter PHI from Request Logs
Add PHI-containing parameter names to config.filter_parameters in config/application.rb. At minimum, filter: :query, :birth_date, :insurance_nr, :family_name, :given_name, :finding_text. Also enable config.log_tags to associate requests with user IDs for traceability.
3
Upgrade Rails and All Dependencies
Rails 3.2 has been end-of-life since 2013 with numerous known CVEs. The upgrade path (3.2 → 4.0 → 4.2 → 5.0 → 5.2 → 6.1 → 7.x) is substantial but necessary. As an interim measure, evaluate which specific CVEs are exploitable in the current deployment context. Replace the CanCan gem with CanCanCan (the maintained fork). Update Devise to current stable. Change Gemfile source to https://rubygems.org.
3
Enable SSL Enforcement and Security Headers
Add config.force_ssl = true to config/environments/production.rb. Add the SecureHeaders gem to set Content-Security-Policy, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, and Strict-Transport-Security. Enable :secure and :httponly for session cookies.
4
Implement a CI/CD Pipeline
Add GitHub Actions (or equivalent) to run the test suite on every push. This is foundational for DORA improvement. The existing test infrastructure (MiniTest + RSpec + Cucumber) provides a starting point. Fix or remove the heavily commented-out test code in test/unit/patient_test.rb. Add Brakeman (static security analysis for Rails) to the CI pipeline to catch new security issues automatically.
4
Add Monitoring, Health Checks, and Alerting
Add a /health endpoint returning 200 OK when the application is operational. Integrate an exception tracker (Sentry or Honeybadger). Add application performance monitoring. Configure log shipping to a tamper-resistant destination (centralized log management). These improvements directly address the DORA MTTR metric and are required for a production medical application.
5
Evaluate Encryption at Rest (Long-Term)
Field-level encryption with the lockbox gem (Rails 7+) or attr_encrypted is the recommended long-term approach for PHI fields. In the near term, ensure the database server uses full-disk encryption (e.g., dm-crypt/LUKS on Linux) and that the uploads directory is on an encrypted volume. Document the encryption status for Swiss DSG/nDSG compliance reporting.
