CyDoc is a Swiss medical practice management system built on Ruby on Rails. It implements the full lifecycle of a doctor's office: patient registration, consultation sessions, clinical case management (TARMED tariff system), invoice generation, insurance policy tracking, and double-entry bookkeeping. The codebase reflects Swiss healthcare regulatory requirements including KVG/UVG insurance law, ESR payment slips, and Covercard smart-card integration.

Core Domain Areas
♥ Patient Management
Patient is the central entity. Patients have VCard-based contact info (via the has_vcards gem), belong to a Doctor, carry InsurancePolicy records (KVG/UVG law types), and accumulate Treatment chains. Sphinx full-text search indexes name, address, and case numbers. Covercard smart-card integration enables real-time insurance verification.
✹ Clinical Workflow
Treatment → Session → ServiceRecord is the core clinical chain. A treatment represents a medical episode (reason: Krankheit/Unfall/Mutterschaft/etc). Sessions are individual consultations with AASM state machine (active → charged). ServiceRecords are line items using the TARMED tariff system with MT (medical time) and TT (technical time) tax points.
₣ Billing & Accounting
Invoice aggregates ServiceRecords from active sessions. The Tiers system (garant/payant/soldant) handles Swiss insurance billing modalities. ESR (Einzahlungsschein mit Referenznummer) payment slip generation is built in. Double-entry bookkeeping via Booking records with debit/credit accounts. Reminder workflow: booked → reminded → 2x/3x reminded → encashment.
❖ Tariff System
TariffItem uses STI with subtypes: TarmedTariffItem (main Swiss tariff), LabTariffItem (Analyseliste), DrugTariffItem, MigelTariffItem, PhysioTariffItem, FreeTariffItem. Drug catalog imported from Swissmedic: DrugProduct → DrugArticle → DrugPrice. TariffPrices are tagged with reason and unit type for date-range pricing.
Key Architectural Decisions
Person/STI hierarchy: Person is the base class for Doctor, Employee, and others via Rails STI on the people table. VCards are associated polymorphically, following the vCard RFC standard via the has_vcards gem.
Multi-tenancy via Tenant: A Tenant wraps an Office/Person and owns configurable settings (printer names, account codes, invoice grace periods). All controller access goes through current_tenant.
CanCan authorization: AuthorizedController (base for all controllers) uses CanCan with role-based abilities. Only admin and sysadmin roles exist — all users with those roles get can :manage, :all.
InheritedResources pattern: Controllers inherit from AuthorizedController < InheritedResources::Base, auto-generating standard CRUD. Custom actions (print, remind, covercard_update) extend this base.
Sphinx search: Patient uses thinking-sphinx with a delta index strategy. Queries parse date formats (dd.mm.yy) and case numbers (NN/NNNNN) before hitting the index.
Prawn PDF generation: Invoice documents (patient letters, insurance recipes, reminder letters, recall notices) are generated via Prawn with custom layouts in app/prawn/. CUPS integration allows direct printing to network printers.

Notable File Observations
invoice.rb is the largest model at ~645 lines. It contains state machine logic, ESR/payment slip generation, bookkeeping integration, PDF rendering, and dunning (reminder) lifecycle — a sign of the billing domain's complexity.
Two form DSLs coexist: both formtastic and simple_form are present, suggesting an in-progress migration between them during the app's evolution.
Dual i18n locale strategy: Primary is de-CH (Swiss German), with German date parsing (dd.mm.yyyy format) injected globally via initializer. The app was built for Swiss medical practice.
L1 — Module Architecture
CyDoc's architecture is organized into six logical modules, each with clear responsibility boundaries. Controllers follow the InheritedResources pattern extending a shared AuthorizedController base. Models are grouped by domain concern.
Identity & Auth Module
Person Doctor Employee User Role Ability Tenant
Person is the STI base on the people table, extended by Doctor and Employee. User links to any Person via polymorphic object_id/object_type. Devise handles authentication; CanCan enforces role-based access via Ability. Tenant provides multi-practice configuration.
Patient Module
Patient InsurancePolicy Recall Appointment Attachment
Patient holds demographics, belongs to a doctor, carries VCard contact info, and has insurance policies. InsurancePolicy links to KVG/UVG/etc insurers with validity dates. Recall manages preventive care reminders. Appointment tracks scheduled visits. Sphinx indexes full patient data for fast lookup.
Clinical Module
Treatment Session MedicalCase Diagnosis ServiceRecord Law
Treatment is a medical episode with reason (Krankheit/Unfall/etc) and law reference. Session is a single consultation with AASM state machine. MedicalCase (STI: anamnesis, consultation, laboratory, prescription, therapy, attestation) documents clinical findings. ServiceRecord captures individual TARMED tariff positions rendered during a session.
Tariff Catalog Module
TariffItem TarmedTariffItem LabTariffItem DrugTariffItem TariffPrice DrugProduct DrugArticle VatClass
TariffItem is STI base for all billable items. TARMED is the primary Swiss medical tariff: codes have MT (medical) and TT (technical) tax point values, priced by TariffPrice records tagged with reason and unit type. Drug catalog (DrugProduct → DrugArticle → DrugPrice) is imported from Swissmedic. TariffItemGroup bundles related items for quick entry.
Billing & Accounting Module
Invoice Tiers TiersGarant TiersPayant Booking InvoiceBatchJob ReturnedInvoice EsrRecord
Invoice is the core billing artifact, linking Treatment to ServiceRecords via Tiers (billing mode). Swiss billing modalities: TiersGarant (patient pays, claims insurer), TiersPayant (direct insurer billing), TiersSoldant. ESR (Einzahlungsschein) payment slips with check digit calculation. Booking provides double-entry accounting. InvoiceBatchJob handles bulk printing runs.
Administration & Import Module
Insurance Import Importer PostalCode Account Office
Insurance stores Swiss insurer registry with EAN party IDs and BSV codes. Import logs data import operations (TARMED catalog updates, insurer list refreshes, drug catalog from Swissmedic). PostalCode uses the swissmatch gem for Swiss address autocomplete. The bookkeeping sub-module manages the chart of accounts and booking templates.

L3 — Data Flow
CyDoc has two primary data flow pipelines: the Clinical Workflow (patient visit → session → service records) and the Billing Pipeline (invoice creation → booking → ESR payment). These pipelines share the ServiceRecord as the connective tissue.
Clinical Workflow Pipeline
1. Patient Lookup
Sphinx full-text search by name, DOB, or case number. Covercard smart-card lookup via code scan. Patient.by_text(query)
→
2. Open Treatment
Select active Treatment (Krankheit/Unfall/etc) or create new one with Law record. Referrer doctor optional.
→
3. Create Session
Consultation session: duration_from/to, remarks. AASM state: active. Session belongs_to Treatment, touches it on save.
→
4. Add Service Records
Select TariffItems by code or text search. TariffItem.create_service_record(session) calculates MT/TT tax points and unit prices from TariffPrice (tagged by reason + date).
→
5. Document Cases
Add MedicalCase records (anamnesis, consultation findings). Link Diagnoses (ICD/freetext). These feed Treatment.valid_for_invoice?
Billing & Payment Pipeline
1. Invoice Creation
Invoice.create_from_treatment(params, treatment). Collects active session ServiceRecords. Validates patient, treatment, service records.
→
2. Tiers Setup
TiersGarant/TiersPayant defines billing mode. Sets biller, provider, insurance, patient, referrer, employer relationships.
→
3. Book & Print
invoice.book creates Booking debit/credit entries. State: prepared → booked → printed. Prawn generates patient letter + insurance recipe PDFs. CUPS or browser PDF.
→
4. ESR Payment
ESR9 reference code encodes patient_id + invoice_id + esr_id. ESR file uploaded, parsed into EsrRecord. Matched to Invoice → Booking (payment) created.
→
5. Dunning / Write-off
Overdue invoices: booked → reminded → 2xreminded → 3xreminded → encashment → written_off. Each level generates a Booking (reminder fee) and prints a reminder letter. ReminderBatchJob handles bulk runs.

Doctor creates Recall (due_date, remarks). Patient gets preventive care reminder. State: open.
→

recall.send_notice generates Prawn recall letter PDF. Printed via CUPS or downloaded. sent_at timestamp recorded.
→
Appointment Booked
Appointment links back to Recall. date, duration_from/to, state tracked. obey! / accept! transitions.
→

Patient attends, new Treatment/Session created. Recall state updated to completed. Clinical pipeline resumes.

Billing System — Swiss Healthcare Specifics
CyDoc implements the Swiss TARMED (Tarif Medical) billing standard, ESR payment slips, and multi-modality insurance billing (KVG/UVG/MV/IV laws). This is the most complex part of the system and the core value proposition.
TARMED Tariff System
◆ Tax Point Architecture
Every billable service is a ServiceRecord with two independent value channels:

MT (Medical Time / Arzt-Leistung): amount_mt × unit_factor_mt × unit_mt
TT (Technical Time / Technische Leistung): amount_tt × unit_factor_tt × unit_tt

Unit prices (unit_mt, unit_tt) are looked up from TariffPrice records tagged with reason:Krankheit (or Unfall/Mutterschaft) and unit:medical (or technical), filtered by valid_from date.
◆ Tariff Subtypes
tariff_type field identifies the billing system:
001 = TARMED (main medical tariff)
317 = Analyseliste (lab tests)
400 = Drug catalog (Spezialitätenliste)
432 = MIGEL (medical aids)
999 = Free tariff (Eigenleistung)

The TARMED catalog is imported from the official MSSQL database via the tarmed/ model namespace. Drug catalog from Swissmedic CSV files.
Insurance Billing Modalities (Tiers)
TiersGarant (patient pays)
Doctor sends invoice to patient. Patient is responsible for paying. Patient then claims reimbursement from insurer. Most common in Swiss basic insurance (KVG). ESR payment slip sent to patient.
TiersPayant (insurer pays)
Doctor sends invoice directly to insurance company. Used for supplemental insurance (VVG), accident insurance (UVG), military insurance (MV), disability insurance (IV). XML invoice (SumEx format via Nokogiri) generated for electronic submission.
TiersSoldant
Intermediate clearing house billing. Less common, used for specific cantonal arrangements.
ESR Payment Slip System
✓ ESR Reference Number Generation
The ESR9 reference number uniquely identifies an invoice:

esr_id (6 chars) + patient_id (13 digits, zero-padded) + invoice_id (7 digits) = 26-char base

A Luhn-style check digit is appended using the esr9_table (mod 10 recursion).

Final format: grouped in 5-digit blocks from right, stripped of leading zeros. Compatible with Swiss bank clearing system.
✓ ESR File Processing
ESR files (bank payment files) are uploaded via CarrierWave (EsrFileUploader). Each file contains multiple EsrRecord lines with:
bank_pc_id, reference, amount, payment_date, value_date, reject_code

The bookkeeping controller matches ESR records to invoices by reference, creates Booking entries, and transitions invoice state to paid when due_amount ≤ 0.
Double-Entry Accounting
∥ Booking Ledger
Built on the has_accounts gem. Key account codes from Tenant settings:
invoices.profit_account_code — revenue account
invoices.balance_account_code — accounts receivable
invoices.extra_earnings_account_code — overpayment

Invoice booked: DR profit_account / CR balance_account
Payment: DR balance_account / CR bank_account
Cancellation (Storno): reverse original + write off remainder

Schema Evolution Timeline
The migration numbering reveals two development eras: 001–031 (sequential, no timestamps) represent the 2007–2008 initial build phase. Starting with 20080827, migrations become timestamped, reflecting the switch to standard Rails tooling and indicating when the team adopted proper version control discipline. The jump from consultation → session (20090604) was a significant rename that touched 4 tables and required data migration.
