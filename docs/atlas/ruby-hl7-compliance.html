ruby-hl7 is a mature Ruby library for parsing and generating HL7 v2.x healthcare messages. As of version 1.4.0 (January 2025), the project has resumed active development after a five-year near-dormancy (2019–2024), with 51 commits in Q4 2024 alone. The library covers 35+ segment types, has 152 test cases with broad spec coverage, and recently added GitHub Actions CI across Ruby 3.0–3.4+head.

From a HIPAA perspective, this is a library that parses messages containing PHI — patient names, SSNs, diagnoses, lab results, medications — and transforms them into Ruby objects. The library itself does not log or persist data, which is the correct baseline. However, three critical risks exist: (1) the example proxy server transmits MLLP over plaintext TCP with no TLS option; (2) deprecation warnings from PID and PV2 segments emit to STDERR, creating a PHI-adjacent channel in production logs; and (3) the eval()-based segment class lookup accepts arbitrary segment names from untrusted input, creating a potential code injection vector if improperly handled upstream.

However, the gap between code being merged to master and an actual gem release is often measured in months to years. The release process requires manual rake gemspec and RubyGems push steps, with no automation:

⬡ HIPAA Compliance Assessment

Framing: ruby-hl7 is a library, not a covered entity or business associate directly. However, any application using it to process real patient messages becomes a HIPAA actor. The library's design choices directly determine whether PHI can leak, persist, or be exposed inadvertently. The analysis below evaluates the library as a PHI processor.

A. Data Handling — Logging & Persistence

No Persistence or Structured Logging of PHI PASS

The library contains no calls to Logger, Rails.logger, file I/O, or database operations in the core parsing path. Parsed PHI exists only in instance variables (@elements array on each Segment object) within the calling process's memory space.

# lib/segment_fields.rb — PHI is stored only in @elements
def write_field(name, value)
  idx, field_blk = field_info(name)
  return nil unless idx
  # ...expand @elements array if needed...
  @elements[idx] = value.to_s  # stored only in memory
end

There is no caching layer, no class-level storage of parsed data, and no persistence mechanism. PHI is scoped to the lifetime of the HL7::Message object and its child Segment instances.

Deprecation Warnings Emit to STDERR — PHI-Adjacent CONCERN

Three deprecation warnings use Ruby's warn method, which emits to STDERR. While the warnings themselves do not include PHI field values (only method names), they fire during normal PHI processing — specifically when accessing deprecated PID and PV2 fields on messages containing patient data. In production environments where STDERR is captured to log aggregators (Splunk, Datadog, CloudWatch), these warnings create noise that could appear alongside PHI in correlated log entries.

# lib/segments/pid.rb:55
def country_code
  warn "DEPRECATION WARNING: PID-12 is defined as 'county_code'; " \
       "the 'country_code' alias is retained for backwards compatibility only."
  county_code  # <-- returns PHI value; warn fires before this
end

More critically, if any future deprecation warning were inadvertently modified to include field content (e.g., for debugging), it would instantly become a HIPAA issue. The pattern is fragile.

B. Memory Safety — PHI Retention

PHI Not Cleared from Memory After Use GAP

Ruby's garbage collector handles memory, but Ruby does not zero-fill memory on object deallocation. PHI stored in @elements strings will remain in process memory until GC reclaims and reallocates those pages. This is a fundamental Ruby limitation, not a library bug. However, the library provides no mechanisms to assist callers in clearing PHI early:

  • No clear! or scrub! method on HL7::Message or Segment
  • No documentation guidance on PHI lifecycle management
  • No freeze-after-use pattern to prevent accidental PHI modification after processing

For long-running processes (Sidekiq workers, Puma servers) processing many messages, PHI from earlier messages may linger in memory for extended periods.

Class-Level Singleton State: HL7.configuration CONCERN

The HL7.configuration object is a module-level singleton (@configuration ||= Configuration.new). This is process-global state, shared across all threads and all message parsing operations. While the current configuration only controls empty_segment_is_error, the pattern creates a class-level object that persists for the process lifetime. In multi-tenant environments where different callers need different configuration, this is a risk. The singleton also means configuration changes in one part of a multi-threaded application affect all concurrent PHI processing.

# lib/ruby-hl7.rb — process-global singleton
module HL7
  def self.configuration
    @configuration ||= Configuration.new  # shared across ALL threads
  end
end

C. Transmission Security — MLLP / TLS

MLLP Support Without TLS — Critical PHI Risk CRITICAL

The library provides full MLLP framing support via to_mllp and parses MLLP-wrapped input in parse_string. MLLP (Minimal Lower Layer Protocol) is the standard transport for HL7 v2.x messages over TCP. The library correctly implements the MLLP envelope (\x0b...\x1c\r), but there is zero TLS/SSL anywhere in the library or its examples.

# lib/message.rb — MLLP framing, no TLS context
def to_mllp
  pre_mllp = to_hl7
  "\v#{pre_mllp}\u001C\r"  # VT + content + FS + CR — raw bytes, no encryption
end

# lib/message_parser.rb — MLLP parsing, accepts plaintext
def parse_string(instr)
  post_mllp = instr
  if /\x0b((:?.|\r|\n)+)\x1c\r/ =~ instr
    post_mllp = $1  # strips MLLP bytes — no TLS enforcement
  end
  HL7::MessageParser.split_by_delimiter(post_mllp, @delimiter.segment)
end
Proxy Server Example: Plaintext TCP, No TLS, Broken Implementation CRITICAL 
# examples/proxy_server.rb — MULTIPLE CRITICAL ISSUES
srv = TCPServer.new(PORT)        # ❌ plaintext TCP — no TLS
loop do
  sok = srv.accept
  Thread.new(sok) do |my_socket|
    my_socket.readlines           # ❌ result discarded — raw_input is nil!
    msg = HL7::Message.new(raw_input) # ❌ raw_input undefined — this crashes
    puts format("forwarding message:\n%s", msg.to_s)  # ❌ PHI to STDOUT
    soc = TCPSocket.open(target_ip, target_port)  # ❌ plaintext downstream
    soc.write msg.to_mllp         # ❌ PHI over plaintext TCP
    soc.close
  end
end

The proxy server example has five distinct HIPAA and correctness issues: plaintext transport in both directions, PHI logged to STDOUT via puts, a broken implementation (reads socket to void, then references undefined raw_input — would crash immediately), and no authentication or access control. While labeled "example," beginners copy example code into production.

D. Input Validation & Injection Risks

eval() on Segment Name from Untrusted HL7 Input CRITICAL

The segment class lookup uses Ruby's eval() to resolve segment names extracted directly from parsed HL7 messages:

# lib/segment_generator.rb:42-49
def get_segment_class
  segment_to_search = @seg_name.to_sym
  segment_to_search = @seg_name if RUBY_VERSION < "1.9"

  if HL7::Message::Segment.constants.index(segment_to_search)
    eval(format("HL7::Message::Segment::%s", @seg_name))  # ❌ eval on parsed input!
  else
    HL7::Message::Segment::Default  # fallback — safe path
  end
end

The @seg_name value comes from parsing the first field of each segment line in the incoming HL7 message. The eval is guarded by HL7::Message::Segment.constants.index(segment_to_search) — this check verifies the segment name is an existing constant before evaluating it. This means the guard prevents arbitrary code injection in practice: only pre-defined segment names like MSH, PID, etc. can pass the guard.

However, this is a defense-by-coincidence, not by design. The guard uses .constants.index() which checks Ruby's constant table — if any future code dynamically defines a constant matching a crafted segment name, the guard could be bypassed. The safer alternative is a whitelist hash lookup or const_get with an explicit allowlist. The use of eval on parsed input is an architectural smell that should be refactored regardless of current exploitability.

No Input Size Limits — DoS via Unbounded Parsing HIGH

The parser imposes no size limits on incoming messages. A malformed or adversarial HL7 message with thousands of segments, extremely long field values, or deeply nested children would be parsed in full, consuming unbounded memory and CPU. The split_by_delimiter call uses Ruby's String#split with -1 (unlimited splits):

# lib/message_parser.rb:51
def self.split_by_delimiter(element, delimiter)
  element.split(delimiter, -1)  # -1 = unlimited splits, no size guard
end

The performance test (spec/speed_parsing_spec.rb) uses lotsunknowns.hl7 as the "large message" test case — but this is a finite, known test file. There is no fuzzing or property-based testing with adversarial inputs. Healthcare message brokers often accept HL7 from external labs or EHR systems, making DoS via malformed input a realistic threat.

Escape Character Handling — Limited Validation MEDIUM

HL7 v2.x uses escape sequences (\F\, \S\, \R\, \E\, \T\) for encoding special characters within field values. The library's parser splits on delimiters but does not validate or unescape these sequences. A field value containing delimiter characters embedded via escape sequences will be incorrectly split, potentially causing field misalignment in downstream processing. This is both a correctness bug and a potential data integrity risk for PHI fields (e.g., a patient name containing a pipe character escaped per HL7 spec).

method_missing on Unknown Fields — Silent Data Loss Risk HIGH

The method_missing implementation in Segment handles both named field aliases and numeric element accessors (e0, e1, etc.). For completely unknown method names, it falls through to super, which raises NoMethodError. This is correct.

However, the Segment::Default class — the catch-all for unknown segment types — stores all data but provides no named accessors. Any PHI in an unknown segment type (e.g., a custom Z-segment containing patient identifiers) is preserved in @elements but completely inaccessible by name. Consuming code must know to use e0, e1, etc. numeric accessors, or it will silently miss the data. There is no warning that a segment fell through to Default.

# lib/segment_generator.rb:46-50
if HL7::Message::Segment.constants.index(segment_to_search)
  eval(format("HL7::Message::Segment::%s", @seg_name))
else
  HL7::Message::Segment::Default  # silent fallback — no log, no warning
end

In a batch processing context, if a message contains a Z-segment with a patient identifier that the consuming application expects to parse by name, the silent Default fallback means the identifier is silently dropped from the application's view — a PHI integrity risk.

E. Audit Trail & Access Hooks

No Audit Hooks or PHI Access Tracking Mechanism GAP

HIPAA's Technical Safeguards (45 CFR §164.312(b)) require audit controls that record and examine activity in systems containing PHI. The library provides no hooks, callbacks, or observer patterns that consuming applications could use to record which PHI fields were accessed, by which process, at what time. Every field access via read_field is silent and untracked:

# lib/segment_fields.rb:101
def read_field(name)  # no audit hook — access is untracked
  idx, field_blk = field_info(name)
  return nil unless idx
  return nil if idx >= @elements.length
  ret = @elements[idx]
  ret = field_blk.call(ret) if field_blk
  ret
end

There is also no documentation guiding library users on how to implement compliant audit logging in their consuming applications. This is an educational gap for healthcare developers who may not realize that parsing a PID segment and reading social_security_num constitutes a PHI access event that should be logged under HIPAA.

F. Batch Parsing — PHI Leakage Between Messages

Batch Parsing: Correct Isolation Architecture PASS with NOTE

The batch parser in message_parser.rb correctly yields each message as a separate string to the caller via a block. Each message is then parsed into a fresh HL7::Message instance with its own @segments array. There is no shared state between messages in a batch at the library level.

# lib/message_parser.rb
def parse_batch(batch)
  match.post_match.split("\rMSH").each do |_msg|
    yield "MSH#{_msg}"  # each message is a separate string — correct isolation
  end
end

Note: The batch parser performs string splitting on the raw batch in memory before yielding. For very large batches (e.g., thousands of lab results), the entire batch string is held in memory simultaneously. There is no streaming parser option. This is a memory pressure concern for high-volume healthcare integrations, and the full batch in memory means all patients' PHI exists in the process simultaneously.

G. HIPAA BAA Implications & Guidance

No Guidance for Compliant Use in Healthcare Contexts GAP

The library's README, LICENSE (MIT), and documentation contain zero mention of HIPAA, PHI, or compliant integration patterns. Healthcare developers using this library to build systems that process real patient data are given no guidance on:

  • That the library processes PHI and downstream applications inherit HIPAA obligations
  • How to implement audit logging at the application layer
  • That MLLP transport requires TLS in compliant deployments
  • That the proxy server example should not be used without TLS in production
  • Memory handling considerations for PHI in long-running processes

The MIT license explicitly disclaims warranties and liability. In a healthcare context, this means organizations using ruby-hl7 bear full compliance responsibility, but they are given no tools to discharge it.

◆ Key Findings Summary

FIND-01 eval() on Parsed HL7 Segment Names Critical Injection Risk

In lib/segment_generator.rb:47, eval(format("HL7::Message::Segment::%s", @seg_name)) executes Ruby code constructed from a value read directly from a parsed HL7 message. The guard (constants.index()) prevents current exploitation but the pattern is architecturally unsafe. Future changes to constant loading or dynamic class creation could bypass it.

File: lib/segment_generator.rb L42–51 | Recommendation: Replace with const_get against an explicit allowlist, or a pre-built Hash map of segment name → class.

FIND-02 Proxy Example: Plaintext PHI Transmission + Broken Code Critical PHI in Transit

examples/proxy_server.rb opens a plaintext TCPServer (no TLS), logs PHI to STDOUT via puts msg.to_s, and contains a bug (raw_input is undefined — the code would crash on line 20). The example demonstrates the exact anti-pattern that HIPAA prohibits: unencrypted PHI transmission over a network.

File: examples/proxy_server.rb | Recommendation: Remove or replace with a TLS-based example using OpenSSL::SSL::SSLServer, and remove puts of message content.

FIND-03 MLLP Without TLS — Library Promotes Plaintext PHI Transport Critical HIPAA §164.312(e)(1)

The to_mllp() and MLLP-parsing regex in parse_string() implement raw MLLP without any TLS wrapping capability. The library provides no OpenSSL integration, no TLS context, and no documentation that MLLP transport must be TLS-encrypted in HIPAA-compliant deployments. HIPAA Technical Safeguards require encryption of PHI in transit (§164.312(e)(2)(ii)).

Files: lib/message.rb:174–177, lib/message_parser.rb:61–65 | Recommendation: Add HIPAA notice to README; provide TLS-wrapped MLLP example; optionally add optional TLS transport helper.

FIND-04 No Input Size Limits — DoS via Unbounded Message Parsing High Availability Risk

No maximum message size, segment count, or field length limits exist. split_by_delimiter uses -1 (unlimited). Adversarially crafted messages or corrupted feeds could cause unbounded memory allocation. Healthcare HL7 feeds often originate from multiple external systems that may send malformed or adversarial data.

File: lib/message_parser.rb:51 | Recommendation: Add configurable limits to HL7::Configuration: max_message_size, max_segment_count, max_field_length.

FIND-05 Silent Default Segment Fallback — PHI Data Loss Risk High Data Integrity

Unknown segment types silently fall through to Segment::Default with no warning, log, or callback. Consuming applications have no way to know that PHI-containing segments were not parsed with named accessors. Z-segments (custom extensions) frequently carry patient identifiers in healthcare integrations.

File: lib/segment_generator.rb:49 | Recommendation: Add optional warning configuration (warn_on_unknown_segment: true) and/or provide a callback hook for unknown segments.

FIND-06 Deprecation Warnings Emit to STDERR During PHI Processing High Log Hygiene

PID country_code, id_readability_code and PV2 military_non_availiOCbility_code aliases emit warn to STDERR. While current values don't include PHI, the STDERR output occurs during PHI processing, creating correlation risk in aggregated logs. Production systems should not call deprecated methods, but legacy integrations often do.

Files: lib/segments/pid.rb:55,61,67,73, lib/segments/pv2.rb:56,63 | Recommendation: Route deprecation warnings through HL7.configuration.logger (add logger support) or suppress in production via configuration.

FIND-07 No PHI Memory Clearing Mechanism Medium Memory Hygiene

PHI stored in @elements string arrays is not zero-filled or actively cleared after use. For long-running processes, PHI from processed messages may linger in GC-managed memory. The library provides no clear!, scrub!, or secure disposal API.

File: lib/segment_fields.rb | Recommendation: Add a HL7::Message#clear! method that replaces all element values with empty strings, and document the memory lifecycle.

FIND-08 HL7 Escape Sequence Handling Absent Medium Data Integrity

HL7 v2.x specifies escape sequences (\F\ for field separator, \S\ for subcomponent separator, etc.). The library splits on literal delimiter characters only. Field values containing escaped delimiters will be incorrectly split, causing PHI field misalignment. Patient names or addresses containing pipe characters encoded per spec would be corrupted.

Files: lib/message_parser.rb, lib/segment_fields.rb | Recommendation: Implement HL7 escape/unescape support per v2.x specification §2.10.

FIND-09 No Audit Trail Hooks for PHI Access Medium HIPAA §164.312(b)

HIPAA Technical Safeguards require audit controls. The library provides no mechanism for consuming applications to intercept PHI field access events. Every call to pid.social_security_num, pid.patient_name, etc. is invisible to audit infrastructure. No documentation guides developers on implementing compliant audit logging.

File: lib/segment_fields.rb:101-110 | Recommendation: Add optional audit callback hook: HL7.configure { |c| c.on_phi_access = ->(segment, field, value) { audit_log(segment, field) } }.

FIND-10 eval() in Segment Block Initializer via Binding Medium Code Quality

The block-based segment initializer uses three eval() calls against the block's binding to inject method_missing behavior. While this is a controlled use of eval (not operating on external input), it is an advanced Ruby pattern that creates invisible behavior in the calling scope and complicates security audits.

# lib/segment.rb:54-63
callctx = eval("self", blk.binding, __FILE__, __LINE__)
# ...sets up __seg__ and method_missing on caller's binding...
eval(METHOD_MISSING_FOR_INITIALIZER, blk.binding)
yield self
eval("class << self; remove_method :method_missing;end", blk.binding, __FILE__, __LINE__)

File: lib/segment.rb:54-63 | Recommendation: Refactor to use explicit block DSL without binding manipulation. This pattern is used only in the block-form constructor which is non-standard API.

FIND-11 SimpleCov Not Enforced in CI — No Coverage Gate Medium DORA Quality

SimpleCov is configured in spec_helper.rb but only activated when ENV["COVERAGE"] is set. The CI workflow (test.yml) does not set this variable, so coverage is never measured in CI. There is no minimum coverage threshold enforced.

Files: spec/spec_helper.rb:4-10, .github/workflows/test.yml | Recommendation: Enable SimpleCov in CI unconditionally and set a minimum coverage threshold (e.g., 90%).

FIND-12 No Security Disclosure Process or SECURITY.md Low Supply Chain

There is no SECURITY.md, no GitHub Security Advisories configuration, no Dependabot, and no responsible disclosure process documented. For a library used in healthcare applications, this is a meaningful gap.

Recommendation: Add SECURITY.md with a private disclosure contact, enable GitHub Security Advisories, and configure Dependabot for dependency updates.

◈ Recommendations

For Library Maintainers

REC-01 — Eliminate eval() in get_segment_class() [Priority: High]
Replace eval(format("HL7::Message::Segment::%s", @seg_name)) with HL7::Message::Segment.const_get(@seg_name) inside a rescue block, or pre-build a frozen Hash mapping segment name strings to class objects at load time. This removes the eval entirely and is faster (hash lookup vs. eval + constant resolution).
REC-02 — Remove or Completely Rewrite the Proxy Server Example [Priority: Critical]
The example is broken (undefined raw_input), logs PHI to STDOUT, and uses plaintext TCP. Either remove it or replace it with a TLS-using example that demonstrates OpenSSL::SSL::SSLServer, removes all puts of message content, and includes a HIPAA compliance comment.
REC-03 — Add HIPAA Usage Notice to README [Priority: Critical]
Add a dedicated "Healthcare / HIPAA Compliance" section to the README covering: (1) MLLP must be used over TLS in HIPAA environments; (2) applications must implement audit logging; (3) PHI lifecycle management in consuming code; (4) the library provides no encryption and callers are responsible for transport security.
REC-04 — Add Configurable Input Size Limits [Priority: High]
Add to HL7::Configuration: max_message_bytes (default: unlimited for now, recommend 1MB), max_segment_count, max_field_bytes. Raise HL7::ParseError when exceeded. This protects against DoS from malformed inputs.
REC-05 — Add PHI-Aware Audit Hook [Priority: High]
Add an optional audit callback to HL7::Configuration: config.phi_access_callback = ->(segment_type, field_name) { MyAuditLog.record(segment_type, field_name) }. When set, call this from read_field. Mark PHI fields with a metadata flag in add_field options so only sensitive fields trigger the callback.
REC-06 — Enable SimpleCov in CI with a Coverage Minimum [Priority: Medium]
In .github/workflows/test.yml, add env: COVERAGE: "true" and in spec_helper.rb configure SimpleCov.minimum_coverage 90. Upload coverage reports as workflow artifacts.
REC-07 — Add SECURITY.md and GitHub Security Advisories [Priority: Medium]
Create SECURITY.md with a private disclosure email or GitHub Advisory process. Enable Dependabot for development dependency updates. Configure Bundler Audit in CI to catch vulnerable gem dependencies.
REC-08 — Add HL7::Message#clear! for PHI Disposal [Priority: Medium]
Add a clear! method on HL7::Message and Segment that replaces all element values with empty strings (@elements.fill("")) and clears @segments_by_name. Document the PHI lifecycle and when to call it. In Ruby, strings cannot be zero-filled at the OS level, but clearing references is best effort and reduces risk window.
REC-09 — Warn or Callback on Unknown Segment Types [Priority: Medium]
Add a configuration option config.unknown_segment_handler = ->(seg_name) { Rails.logger.warn("Unknown HL7 segment: #{seg_name}") }. When Segment::Default is used as fallback, invoke this handler. This enables consuming applications to detect Z-segments containing PHI that require special handling.
REC-10 — Implement HL7 Escape Sequence Decoding [Priority: Low/Medium]
Add escape sequence processing per HL7 v2.x §2.10: \F\→|, \S\→^, \R\→~, \E\→\, \T\→&. Apply during read_field and reverse during write_field. Gate behind a configuration flag for backward compatibility.

For Library Users (Healthcare Integrators)

USER-01 — Never Use MLLP Without TLS [Priority: Mandatory]
Wrap all MLLP connections with TLS using Ruby's OpenSSL::SSL. The library's to_mllp and parse_string work identically over TLS-wrapped sockets. Mutual TLS (mTLS) is strongly recommended for healthcare network connections.
USER-02 — Implement Application-Level Audit Logging [Priority: Mandatory]
Since the library provides no audit hooks, implement logging at the application layer. Log: message receipt timestamp, sending system (MSH.3), message type (MSH.9), patient identifier read operations (PID.3 access), and processing results. Store logs in a HIPAA-compliant audit log system for minimum 6 years.
USER-03 — Set Message Size Limits at the Network Layer [Priority: High]
Until the library adds native limits, enforce size limits before passing input to HL7::Message.new: raise "Message too large" if raw_input.bytesize > 1_048_576.
USER-04 — Handle Segment::Default Explicitly [Priority: High]
After parsing, check for unknown segments: msg.each { |seg| Rails.logger.warn("Unknown segment: #{seg.e0}") if seg.is_a?(HL7::Message::Segment::Default) }. Never assume all PHI was parsed into named fields.
USER-05 — Minimize PHI Object Lifetimes [Priority: Medium]
Process and extract needed fields from HL7::Message objects as quickly as possible, then allow them to go out of scope for GC. Do not cache parsed message objects in application-level caches (Redis, Memcached) — only cache derived, de-identified data.

◉ Final Verdict

ruby-hl7 v1.4.0 — Fit for Purpose with Conditions Conditional

ruby-hl7 is a well-structured, adequately tested, and actively maintained parsing library with a healthy recent trajectory. Its core parsing engine is solid: no internal logging, no persistence, correct MLLP framing, good segment coverage, and a reasonable test suite with real-world HL7 samples including NIST, ARUP, IHC, and Cerner test data.

However, three conditions must be met for use in HIPAA-covered applications:

  1. Mandate TLS at the transport layer. The library's MLLP support must always be used over TLS-encrypted sockets. The proxy server example must be treated as non-functional and potentially dangerous.
  2. Implement application-level audit logging. The library provides no PHI access hooks. Every patient data access must be logged by the consuming application to satisfy HIPAA audit requirements.
  3. Add input validation before passing to the parser. Guard against DoS by enforcing message size limits before calling HL7::Message.new.

With these conditions met, ruby-hl7 can serve as the HL7 parsing layer in a HIPAA-compliant application. The responsibility for the remaining compliance controls — encryption, access control, audit logging, minimum necessary PHI access — rests entirely with the consuming application.
